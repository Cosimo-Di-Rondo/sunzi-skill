### 一句话介绍

> AI 写代码越来越强了，但依然是个 **"不评估就开干、硬刚最难部分、一条路走到黑"** 的莽夫。
>
> 本项目给它塞了一本《孙子兵法》——让它学会 **先算、先看、先想，再动**。

《孙子兵法》十三篇，5000 余字，被翻译成 29 种语言，至今仍是西点军校、哈佛商学院的参考教材。它的价值不在于教人打仗，而在于提供了一套**通用的决策与行动框架**。

本项目将这套框架提取为 **9 个可执行的 AI Agent Skill**，每个 Skill 都有明确的触发条件、操作流程、强制输出格式和常见错误表。不是让 AI 背兵法，而是让它**真正像孙子一样思考**。

> 灵感来自 [qiushi-skill](https://github.com/HughYau/qiushi-skill) 和 [superpowers](https://github.com/obra/superpowers)。

## Quick Start

```bash
# Claude Code — 一行搞定
git clone https://github.com/Cosimo-Di-Rondo/sunzi-skill
cd sunzi-skill && claude --plugin-dir .

# Cursor — 克隆后将目录加入插件路径即可
# Codex — 同上，见 .codex/ 配置
# 通用 — 将 skills/strategic-thinking/SKILL.md 注入 system prompt
```

安装后 AI 会自动加载"兵法思维"路由器。**不需要记命令**，它会根据场景自动判断是否调用下游 Skill。

想手动触发？直接输入 `/strategic-calculation`、`/void-solid-analysis` 等命令即可。

验证安装：`bash tests/validate.sh`（Windows: `powershell tests/validate.ps1`）

## 使用示例

以下是装上孙子 Skill 前后的对比。从"蛮力推进"到"谋定后动"，区别一目了然。

### 示例一：技术选型决策 —— 庙算决策

**你说：** "我想把项目的数据库从 MySQL 迁移到 PostgreSQL，你觉得怎么样？"

<details>
<summary>❌ <b>没有孙子 Skill 的 AI：直接开干</b></summary>

> 好的！MySQL 迁移到 PostgreSQL 是个好主意。以下是迁移步骤：
> 1. 导出 MySQL 数据
> 2. 安装 PostgreSQL
> 3. 使用 pgloader 导入数据
> 4. 修改代码中的数据库连接
> ...

**问题：** 没有评估"该不该迁"，直接告诉你"怎么迁"。万一迁移毫无必要呢？

</details>

<details open>
<summary>✅ <b>装了孙子 Skill 的 AI：先算后战</b></summary>

> 庙算评估表：
>
> 此次庙算评估的是：**MySQL → PostgreSQL 数据库迁移**
>
> | 维度 | 评分 | 理由 |
> |------|------|------|
> | 道（使命对齐） | 中性 | 迁移动机需要明确——是性能瓶颈、功能需求还是团队偏好？ |
> | 天（时机环境） | 劣势 | 项目正在冲刺期，迁移引入的不确定性风险高 |
> | 地（地形条件） | 劣势 | 现有代码深度绑定 MySQL 特性（存储过程、特定函数），迁移成本大 |
> | 将（领导能力） | 中性 | 团队有 PostgreSQL 经验但缺乏大规模迁移经验 |
> | 法（组织流程） | 劣势 | 缺少迁移回滚方案，CI/CD 未覆盖双数据库测试 |
>
> **庙算结论：No-Go**
> **关键理由：** 在项目冲刺期启动基础设施迁移，风险收益比不合理。
> **改善建议：** 明确迁移动机 → 等冲刺结束 → 建立双库并行测试环境后再评估。

</details>

**区别在哪里？** 孙子教的是"多算胜，少算不胜"——先用五事七计做系统评估，算清楚了再动。不是不让你迁，而是告诉你**现在不是迁的时候**。

### 示例二：复杂 Bug 排查 —— 虚实分析

**你说：** "用户反馈页面加载很慢，有时候要 8 秒以上，帮我排查一下。"

<details>
<summary>❌ <b>没有孙子 Skill 的 AI：硬刚最强点</b></summary>

> 让我们来优化一下！
> 1. 首先给数据库查询加索引
> 2. 然后启用 Redis 缓存
> 3. 再做 CDN 加速
> 4. 最后上代码分割...

**问题：** 一股脑列出所有优化方向，没搞清楚瓶颈到底在哪里。可能 80% 的延迟来自一个 N+1 查询，但 AI 让你把所有事情都做一遍。

</details>

<details open>
<summary>✅ <b>装了孙子 Skill 的 AI：避实击虚</b></summary>

> 虚实地图：
>
> 此次虚实分析的对象是：**页面加载慢（8s+）的性能瓶颈**
>
> | 模块 | 虚实 | 分析 |
> |------|------|------|
> | 前端渲染 | 实 | React 组件树较浅，Lighthouse 性能分 92，不是瓶颈 |
> | 静态资源 | 实 | 已启用 gzip，bundle 体积 180KB，合理 |
> | API 响应 | **虚** | `/api/dashboard` 平均响应 6.2s，占总耗时 78% |
> | 数据库查询 | **虚** | 该接口存在 N+1 查询，每次请求触发 47 次 SQL |
> | 网络链路 | 半虚半实 | 国内 CDN 正常，海外节点偶发超时 |
>
> **攻击建议：**
> - **最佳突破口：** `/api/dashboard` 的 N+1 查询 —— 修复它预计可将响应时间从 6.2s 降至 < 500ms
> - **应避开的实点：** 前端渲染优化（投入大、收益小）
> - **集中资源：** 用 eager loading 改写该查询，涉及 2 个文件约 30 行改动
> - **动态检查点：** 修复后重新测量，若仍 > 2s 则分析网络链路

</details>

**区别在哪里？** 孙子说"避实而击虚"——不要平均分配力量，找到那个"虚"的点，集中火力打击。一个 N+1 查询就能解决 78% 的问题，何必大动干戈？

### 示例三：陌生项目上手 —— 知彼知己

**你说：** "我刚接手一个遗留项目，要加一个用户权限系统，不知道从哪下手。"

<details>
<summary>❌ <b>没有孙子 Skill 的 AI：凭空开干</b></summary>

> 好的，我来帮你设计一个 RBAC 权限系统！
> 1. 创建 users、roles、permissions 三张表
> 2. 实现中间件进行权限校验
> 3. 在每个路由上加装饰器
> ...

**问题：** 完全没了解项目现状——万一项目已经有半成品的权限模块呢？万一用的是 NoSQL 呢？万一业务场景根本不需要 RBAC 呢？

</details>

<details open>
<summary>✅ <b>装了孙子 Skill 的 AI：先知后动</b></summary>

> 知彼知己表：
>
> 此次情报调查的目标是：**遗留项目中新增用户权限系统**
>
> **【知彼】—— 关于项目现状的发现：**
> - 核心机制：Express + MongoDB 技术栈，无 ORM，直接使用原生驱动
> - 强项：路由结构清晰，已有 JWT 认证中间件
> - 弱项：无角色概念，所有认证用户权限相同；无测试覆盖
> - 隐含规则：已有 12 个 API 端点在用 `req.user`，改动需向后兼容
>
> **【知己】—— 关于自身的评估：**
> - 可用能力：熟悉 Express 中间件模式，可快速实现
> - 能力缺口：对该项目的业务语义不够了解，不确定哪些操作需要区分权限
> - 硬约束：不能中断现有 12 个端点的正常工作
>
> **【盲区】—— 仍然未知的关键信息：**
> - [ ] 业务上需要几种角色？权限粒度到什么程度？—— 风险等级：**高**
> - [ ] 是否有多租户需求？—— 风险等级：中
>
> **情报结论：**
> - 最佳攻击点：在现有 JWT 中间件上扩展，添加 role 字段，成本最低
> - 最大风险点：不明确业务需要几种角色就开始建表，可能返工
> - **建议：先和需求方确认角色定义，再开始实现。我可以先准备一个角色方案草稿供讨论。**

</details>

**区别在哪里？** 孙子说"知彼知己者，百战不殆"——先搞清楚项目是什么情况、自己能做什么不能做什么，找出盲区，再定方案。不然就是"不知彼不知己，每战必殆"。

### 示例四：方案缺乏亮点 —— 奇正结合

**你说：** "帮我设计一个技术博客的 SEO 优化方案。"

<details>
<summary>❌ <b>没有孙子 Skill 的 AI：教科书方案</b></summary>

> SEO 优化方案：
> 1. 优化 meta 标签
> 2. 添加 sitemap.xml
> 3. 提升 Lighthouse 评分
> 4. 加 alt 标签
> 5. 做好内链...

**问题：** 这些每篇 SEO 文章都在说，完全没有差异化。所有竞争对手也在做同样的事。

</details>

<details open>
<summary>✅ <b>装了孙子 Skill 的 AI：以正合，以奇胜</b></summary>

> **[正] 稳健基本面** — 必须做好的常规项：
> - meta 标签、sitemap、robots.txt、结构化数据 — 这些是入场券
> - Core Web Vitals 达标（LCP < 2.5s, CLS < 0.1）
> - 内链和面包屑导航
>
> **[奇] 创新突破点** — 竞争对手大概率没做的：
> - 每篇技术文章附带**交互式代码演示**（嵌入 StackBlitz/CodeSandbox），用户停留时间翻倍 → 搜索引擎判定为高质量内容
> - 利用文章中的代码块生成**结构化数据（HowTo Schema）**，直接命中 Google 精选摘要
> - 关键文章输出**配套开源工具**，吸引外链反向引用 — 这是竞品博客很难复制的护城河
>
> **组合方式：** [正] 确保搜索引擎能正确抓取和理解内容，[奇] 让内容本身不可替代——正是地基，奇是高楼。
>
> **转化预判：** 交互式演示一旦成为行业标配，下一个奇的方向可能是 AI 驱动的个性化内容推荐。

</details>

**区别在哪里？** 孙子说"以正合，以奇胜"——正面稳住基本盘，用非常规手段创造决定性优势。全正无奇是平庸，全奇无正是冒险。

### 示例五：多 Skill 联动 —— 工作流编排

遇到复杂场景，多个 Skill 会自动串联。你不需要手动编排，路由器会帮你搞定：

```
场景："老板说下周要上线一个推荐系统，但团队没人做过。"

自动触发工作流：
知彼知己 → 庙算决策 → 虚实分析 → 先胜后战

1. [知彼知己] 先调查：推荐系统的技术方案有哪些？团队的技术栈是什么？
                      时间约束是多少？有没有现成的轮子可以用？
2. [庙算决策] 再评估：以当前资源和时间，这事到底能不能做？
                      如果不能 → 给出 No-Go 理由和替代方案
                      如果能 → 进入下一步
3. [虚实分析] 找突破口：全量推荐引擎太重（实），先做基于规则的简单推荐（虚）
                        用最小成本验证效果
4. [先胜后战] 创造条件：先搭建数据采集管道，准备好 A/B 测试框架
                        条件就绪后再真正开发推荐逻辑
```

## 架构

```
知彼知己（总原则）
│
├── 战略层 ── 庙算决策 · 虚实分析
│               做不做？打哪里？
│
├── 方法层 ── 知彼知己 · 奇正结合 · 势节把握
│               怎么看？怎么想？什么时候动？
│
└── 执行层 ── 先胜后战 · 因敌制胜 · 速战速决 · 以迂为直
                怎么准备？怎么变？怎么快？怎么绕？
```

每个 Skill 都包含：

- **触发条件** — 什么时候该用（以及什么时候不该用）
- **方法流程** — 分步操作指南，每步附《孙子兵法》原文
- **操作规程** — 强制输出格式（庙算评估表、虚实地图、奇正方案……），确保 AI 真正执行而不是空谈
- **常见错误表** — 错误 → 孙子的警告 → 正确做法
- **互引关系** — 各 Skill 之间的协作点

### 预置工作流

3 条工作流把多个 Skill 串联成完整的作战方案：

| 场景 | 工作流 | 用人话说 |
|------|--------|---------|
| 新项目，不知从何下手 | 知彼知己 → 庙算决策 → 虚实分析 → 先胜后战 | 先侦察、再评估、找突破口、做准备 |
| 技术难题，需要突破 | 知彼知己 → 虚实分析 → 奇正结合 → 势节把握 → 速战速决 | 摸清情况、找弱点、出奇招、抓时机、快推进 |
| 计划遇阻，需要调整 | 因敌制胜 → 虚实分析 → 以迂为直 → 速战速决 → 因敌制胜(循环) | 灵活应变、重新分析、迂回前进、快速迭代 |

## 与 qiushi-skill 的关系

本项目受 [qiushi-skill](https://github.com/HughYau/qiushi-skill) 启发，采用了相同的 Skill 规范（frontmatter + 操作规程 + hooks 注入）。两者定位互补，完全可以同时安装：

| | qiushi-skill | sunzi-skill |
|---|---|---|
| 思想来源 | 毛泽东思想 | 《孙子兵法》 |
| 核心关注 | 如何分析问题（矛盾、实践、调查） | 如何做出决策（评估、取舍、时机） |
| 总原则 | 实事求是 | 知彼知己 |
| 强项 | 长期复杂任务的方法论 | 竞争环境下的战略决策 |
| 一句话 | 教 AI 做调查研究 | 教 AI 运筹帷幄 |

两者同时装上，你的 AI 就同时拥有了"调查研究"和"运筹帷幄"的能力。效果大于等于两倍。

## FAQ

<details>
<summary><b>Q：这玩意会不会让 AI 变慢？每次都先输出一堆分析？</b></summary>

不会。路由器只在判定"确实需要"时才触发下游 Skill。你让 AI 改个变量名，它不会掏出庙算评估表。孙子也说了，"兵闻拙速"——该快的时候绝不拖。

</details>

<details>
<summary><b>Q：我可以只用其中一两个 Skill 吗？</b></summary>

可以。每个 Skill 都是独立的 Markdown 文件，单独注入 system prompt 也能用。不过推荐至少装上路由器（`strategic-thinking`），让 AI 自己判断什么时候该用什么。

</details>

<details>
<summary><b>Q：支持哪些平台？</b></summary>

Claude Code、Cursor、Codex 均有现成配置。其他平台只要支持 system prompt 注入，就能用——把 `skills/*/SKILL.md` 的内容塞进去就行。

</details>

<details>
<summary><b>Q：和直接在 prompt 里写"请先分析再行动"有什么区别？</b></summary>

区别在于**结构化**。"请先分析"是一句空话，AI 会"分析"两句然后照样蛮干。孙子 Skill 提供的是**强制输出格式**——庙算评估表、虚实地图、知彼知己表——每一步都有模板和校验，确保 AI 真正做了评估而不是装模作样。

</details>

<details>
<summary><b>Q：为什么是《孙子兵法》而不是别的？</b></summary>

因为它是人类历史上第一本系统化的决策方法论，而且足够抽象——它讨论的不是具体战术，而是"如何做决策"的元框架。从战争到商业到产品到代码架构，适用范围极广。况且都 2500 年了还没过时，这本身就是最好的 code review。

</details>
