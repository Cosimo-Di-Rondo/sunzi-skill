# 兵法思维

把这份 skill 当作一个轻量的路由器和校验框架。目标只有两个：

1. 先用 `知彼知己` 约束判断。
2. 只在明确有帮助时调用下游思想武器。

## 总原则：知彼知己

> "知彼知己者，百战不殆；不知彼而知己，一胜一负；不知彼，不知己，每战必殆。"
> —— 孙子《谋攻篇》

**做任何决策之前，先搞清楚两件事：对方（问题/环境/对手）是什么情况，自己（资源/能力/约束）是什么情况。只有双向透彻了解，才能做出正确判断。**

## 核心行为规则

以下是可观测的行为准则：

| 原则 | 可观测的行为 | 违反信号 |
|------|-----------|---------|
| 先算后战 | 动手之前先评估可行性和条件 | 不做评估就直接开干 |
| 知己知彼 | 同时了解问题本身和自身能力边界 | 只看问题不看约束，或只看能力不看需求 |
| 避实击虚 | 选择最有效的切入点而非最明显的 | 硬刚最难的部分，忽略更高效的路径 |
| 因敌制胜 | 根据具体情况灵活调整方案 | 套用模板不考虑实际条件 |

## 调度规则

只有在下列任一条件成立时，才调用下游 skill：

- 任务明显匹配某个 skill 的触发条件
- 当前结果质量会因为该 skill 明显提升
- 你已经遇到阻塞，需要一个明确的方法框架来推进

以下场景通常值得调用对应 skill：

| 你遇到的情况 | 应调用的 skill |
|---|---|
| 要做重大决策，需要系统评估可行性 | `strategic-calculation` 庙算决策 |
| 需要找到系统的薄弱环节或最佳突破口 | `void-solid-analysis` 虚实分析 |
| 信息不足，需要全面了解情况 | `know-both-sides` 知彼知己 |
| 需要在常规方案之外找到创新突破 | `orthodox-unorthodox` 奇正结合 |
| 需要借力、造势或把握时机 | `momentum-timing` 势节把握 |
| 需要先创造条件再行动 | `win-before-fighting` 先胜后战 |
| 情况在变化，需要灵活应变 | `adapt-to-win` 因敌制胜 |
| 任务有时间压力，需要快速推进 | `swift-resolution` 速战速决 |
| 直接路径受阻，需要迂回解决 | `indirect-approach` 以迂为直 |
| 一个任务明确需要多种方法串联 | `workflows` 工作流组合 |

## 不要过度调用

以下情况通常不需要额外加载下游 skill：

- 用户需求非常具体，且执行路径单一明确
- 只是一次性简单输出，不涉及评估、取舍或策略选择
- 宿主平台已经在执行等价流程，此时只需吸收 `知彼知己` 的约束即可

优先原则：

1. 一次只选一个主 skill；确有必要时，再串联第二个
2. 不为了"形式完整"而机械调用全部 skill
3. 如果宿主平台已有更强的系统约束，以宿主规则为准，在其内部吸收本项目的方法论

## 指令优先级

1. **用户的明确指示**（直接请求、项目文档、工作区约束）
2. **宿主平台的系统规则与安全约束**
3. **sunzi skills** 作为补充的方法论框架

## 如何使用思想武器

- **在 Claude Code 中：** 使用 `Skill` 工具调用对应的 skill
- **在支持命令的宿主中：** 使用仓库内 `commands/` 目录中的手动命令入口
- **在其他平台：** 直接读取对应 `skills/*/SKILL.md`

## 核心纪律

兵法讲究"因利制权"——根据实际利害灵活决策。
当某个思想武器能显著改善判断或行动时，用它；当它只会增加形式负担时，跳过它。
